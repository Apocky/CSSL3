# Designing CSSLv3: a no-LLVM shader-and-systems language for the radiance-cascade era

A language that fuses **Slang-class modular shaders, Austral-class linear capabilities, Koka-class effect handlers, and Jai-class comptime metaprogramming** ‚Äî and emits SPIR-V 1.6 and x86-64 machine code directly, without LLVM ‚Äî is technically achievable in 2026, but only if the project ruthlessly scopes its frontend, treats SPIR-T-style structurization as a non-negotiable IR layer, and ships an end-to-end vertical slice (one shader, one CPU kernel, one differentiable pass) within twelve months. The hardest technical risk is not codegen but **the matrix of feature interactions** ‚Äî linear types √ó effect handlers √ó autodiff √ó structured control flow each have published soundness proofs in isolation, and exactly zero published proofs in combination. The hardest project risk, evidenced by a decade of Jai, is **scope drift that delays the first public release past the window of community interest**.

This report synthesizes ten threads of prior art into a concrete design and a 24-month execution plan. It is written for a small team (2‚Äì4 engineers) targeting an Intel Arc A770 desktop and modern x86-64 CPUs, with rendering ambitions in the Sannikov/Gustafsson/Evans tradition: radiance cascades over sparse voxel DAGs with painterly point-splat compositing, all in a single language that compiles host and device code from one source.

## What the prior art actually proves

Three independent strands of evidence converge on the conclusion that CSSLv3 is **feasible but unprecedented in combination**.

**Compiler infrastructure has matured past the LLVM monoculture.** MLIR is now the dominant lowering substrate for GPU DSLs ‚Äî Mojo, Triton, IREE, and CIRCT all sit on it ‚Äî but Slang, naga (wgpu), tint (WebGPU), and Embark's rustc_codegen_spirv all demonstrate that **production-quality SPIR-V can be emitted without ever touching LLVM IR**. Slang in particular ships a hand-rolled SPIR-V backend, a CUDA backend, and a differentiable-IR pass, totaling roughly 400K lines of C++ ‚Äî this is the existence proof that "no-LLVM" does not mean "no production users." The rspirv Rust crate provides a typed builder for SPIR-V module assembly that handles ID allocation, capability tracking, and section ordering; a CSSLv3 backend can build on that pattern in any host language.

**Direct x86-64 emission is well-trodden ground.** Cranelift (used by Wasmtime, Firefox, and now rustc_codegen_cranelift) compiles in roughly 1/10 the time of LLVM with ~30% slower runtime code ‚Äî an acceptable tradeoff for debug builds and an excellent baseline for a research compiler. QBE (Quentin Carbonneaux), the C3 backend, and Zig's self-hosted x86-64 backend (default since Zig 0.14, 2024) all show that **a usable SysV-ABI-compliant code generator is roughly a person-year of work** if you scope to one OS, one ABI, and one ISA extension level (AVX2). Zig's experience is the most directly relevant: their stage1 (C++/LLVM) ‚Üí stage2 (self-hosted, LLVM backend) ‚Üí stage3 (self-hosted, native backend) trajectory took eight years, and **the final removal of LLVM in 2022 was celebrated as freeing the project from a 200 MB dependency that dominated build times**. CSSLv3 should plan for the same staged approach but compress the timeline by skipping a C++ stage1.

**Modern PL features compose, but only in pairs.** Austral demonstrates that linear types + capabilities can be implemented in 600 lines of OCaml. Koka demonstrates that algebraic effect handlers compile to plain C via Xie & Leijen's evidence-passing translation, with Perceus reference counting making them garbage-free. Slang.D (SIGGRAPH Asia 2023) demonstrates that automatic differentiation can be added to a real shading language without breaking existing shaders. Pony's six reference capabilities (`iso`/`trn`/`ref`/`val`/`box`/`tag`) prove that capability subtyping with deny-aliases can be made tractable. **But no shipping language combines linear types, effect handlers, comptime metaprogramming, and SPIR-V codegen** ‚Äî the closest is Slang, which has none of the first three.

## Architecture: a three-IR pipeline anchored on SPIR-T

The single most important design decision is **what intermediate representation sits between the typed AST and the SPIR-V/x86 emitters**. The Embark Rust-GPU team's hard-won lesson, documented across their CHANGELOG and the SPIR-T proposal, is that **lowering directly from an SSA-with-arbitrary-CFG IR (like LLVM IR or rustc's MIR) to SPIR-V is a permanent source of bugs**, because SPIR-V demands structured control flow with explicit `OpSelectionMerge`/`OpLoopMerge` headers before every conditional branch. Their solution, after years of fighting MIR's unstructured CFG, was to insert SPIR-T ‚Äî a typed, structured-by-construction IR with quasi-pointers (`qptr`) for untyped memory ‚Äî as a mandatory pass.

CSSLv3 should adopt the same architecture from day one:

1. **HIR** (typed AST after elaboration) ‚Äî linear-type checking, effect-row inference, capability tracking, autodiff `[Differentiable]` annotations all live here. Keep this layer small: Austral's entire frontend is under 20K lines of OCaml.
2. **MIR** (structured, monomorphic, SSA-within-blocks) ‚Äî the only IR that allows arbitrary control flow. Effect handlers are lowered here via Xie & Leijen's evidence-passing translation, producing plain CPS-free code that targets either SPIR-V or x86 unchanged.
3. **LIR** (target-specific) ‚Äî SPIR-V module builder (rspirv-style) for shaders, register-allocated x86-64 for host. Both consume the same MIR.

The MIR must be **structured by construction** ‚Äî no `goto`, only `if`/`while`/`for`/`switch`/`break`/`continue`/`return` ‚Äî so that SPIR-V emission becomes a syntax-directed walk rather than a structurization pass. This is the opposite of LLVM's design and aligns with Slang's IR. The cost is that some optimizations (irreducible loops, jump threading across merges) become harder; the benefit is that the SPIR-V backend stays under 5K lines instead of the 50K+ that Rust-GPU needed for stackifier-style restructurization.

## SPIR-V emission without LLVM: what the spec actually requires

For the Vulkan 1.3 + Intel Arc A770 target, the SPIR-V module must declare a specific capability set and obey strict ordering rules. **The non-negotiables are**:

- **PhysicalStorageBuffer64 addressing model** for bindless buffer-device-address access ‚Äî required because every load/store through a physical pointer must carry an `Aligned` operand whose value is a multiple of the largest scalar in the pointed-to type (VUID-StandaloneSpirv-PhysicalStorageBuffer64-06314). `OpConvertUToPtr` and `OpConvertPtrToU` must use 64-bit integers (VUID-...-04710). `OpConstantNull` is forbidden for `PhysicalStorageBuffer` pointers; CSSLv3 must emit `OpConvertUToPtr %u64_0` or `OpBitcast` from a `uvec2` instead.
- **VulkanMemoryModel** capability (not GLSL450) ‚Äî gives precise atomic and barrier semantics, and is what Slang and modern engines target.
- **RuntimeDescriptorArray + ShaderNonUniform** for bindless descriptor indexing, with the `NonUniform` decoration applied to every descriptor index expression that isn't provably uniform.
- **GroupNonUniform + arithmetic/ballot/shuffle/quad** subgroup variants for wave intrinsics ‚Äî Arc A770's Xe-HPG architecture has 8/16-wide subgroups depending on shader.
- **Int8/Int16/Float16** plus the `StoragePushConstant16`/`StorageBuffer16BitAccess` capabilities for sub-32-bit storage (16-bit floats and 8-bit ints are not free ‚Äî they require explicit storage capabilities even when arithmetic is supported).
- **DemoteToHelperInvocation** (SPIR-V 1.6 core) replacing the legacy `OpKill`.

Module section ordering is a compiler-wide invariant: Capabilities ‚Üí Extensions ‚Üí ExtInstImports ‚Üí MemoryModel ‚Üí EntryPoints ‚Üí ExecutionModes ‚Üí debug ‚Üí annotations/decorations ‚Üí types/constants/global variables ‚Üí function declarations ‚Üí function definitions. **Forward references are illegal except for `OpTypeForwardPointer`** (used for self-referential structs in `PhysicalStorageBuffer`). The rspirv builder enforces this automatically; a hand-rolled emitter must replicate that discipline or face hours of `spirv-val` debugging.

The validation loop is: emit ‚Üí `spirv-val --target-env vulkan1.3` ‚Üí `spirv-opt -O` (or `-Os`) ‚Üí optionally `spirv-cross` to GLSL/HLSL/MSL for cross-platform debug. For Arc A770 specifically, attaching `NonSemantic.Shader.DebugInfo.100` instructions enables RenderDoc to show CSSLv3 source-line correlation ‚Äî non-negotiable for shader debugging productivity. The `VK_KHR_shader_non_semantic_info` extension allows `debugPrintf` from inside any shader stage, which has eliminated more bugs in the wgpu/naga ecosystem than any other single feature.

A **minimal compute entry point** in disassembled form runs roughly 30 instructions: capability declarations (3‚Äì6), `%glsl_std_450 = OpExtInstImport "GLSL.std.450"`, memory model, entry point with the `LocalSize` execution mode, type declarations for `%void`, `%uint`, `%v3uint`, the input pointer for `gl_GlobalInvocationID`, the function body. CSSLv3's emitter only needs to template this skeleton and fill in the function body from MIR.

## Native x86-64: Cranelift-style or hand-rolled?

The host backend question is binary: **adopt Cranelift as a library, or write a Cranelift-equivalent**. Adopting Cranelift gives you a battle-tested register allocator (regalloc2), a complete SysV ABI, exception/unwind tables, and ~50K lines of saved work. The cost is a Rust dependency and a slightly worse runtime (~30% off LLVM `-O2`). Writing your own ‚Äî the path Zig, QBE, and the C3 compiler took ‚Äî gives full control and a smaller binary, at a cost of roughly 12‚Äì18 person-months for a usable AVX2 baseline.

For CSSLv3, **Cranelift is the right answer for stage 1 and 2**, with a bespoke backend as a stage-3 option if compile-time matters more than runtime. Cranelift's IR (CLIF) is structured similarly enough to CSSLv3's MIR that the lowering is mostly mechanical. The DWARF/CodeView debug-info story is solved. Zig's experience here is instructive: their self-hosted x86-64 backend took years to reach LLVM parity on real workloads, and as of 2024 still falls back to LLVM for `-O3` builds. **Don't fight that battle on day one.**

For the SIMD layer, target AVX2 universally and AVX-512 opportunistically via runtime CPU dispatch ‚Äî the same pattern Highway and ISPC use. Comptime metaprogramming (Jai-style `#run`) is the natural place to generate ISA-specialized variants of hot kernels.

## The PL feature stack: what to take, what to leave

The most defensible feature combination, drawn from the strongest published designs:

**Linear capabilities, Austral-style.** Borretti's 600-line linearity checker is the simplest known implementation of a sound borrow-checker-equivalent. Adopt the linear/capability split directly: every resource (buffer, descriptor set, command encoder, GPU heap allocation) is a linear value that must be explicitly consumed. Austral's "no exceptions, no operator precedence, fully parenthesized expressions" austerity is too extreme for a graphics audience ‚Äî keep operator precedence and add a Rust-style `Result<T, E>` instead. Pony's six-capability matrix (`iso`/`val`/`ref`/`box`/`trn`/`tag`) is more expressive but adds significant cognitive load; **start with three capabilities (`unique`/`shared`/`immutable`) and add more only when concrete use cases demand them**.

**Algebraic effect handlers, Koka-style.** Xie & Leijen's "Generalized Evidence Passing" (ICFP 2021) compiles handlers to plain C with no runtime, and Perceus reference counting (PLDI 2021 distinguished paper) eliminates GC overhead for functional-style code via in-place reuse analysis. The combination gives CSSLv3 the ability to express asynchronous I/O, generators, async/await, exceptions, and dependency injection as **library code** rather than language features. The catch: handlers and linear types interact non-trivially (Borretti has written about how linear types and traditional exceptions are incompatible), and the published combinations are limited. CSSLv3 should restrict effect handlers to non-linear values in v1, and explore linear-effect interaction in v2.

**Comptime metaprogramming, Jai/Zig-style.** Arbitrary functions tagged `#run` (Jai) or `comptime` (Zig) execute during compilation with full read/write access to the AST. This subsumes C++ templates, C macros, build scripts, and code generation in a single mechanism. For a shader language, comptime is **the right answer to shader permutations**: instead of preprocessor `#ifdef` explosion, generate specialized SPIR-V modules at compile time. The risk is the Jai risk: comptime that can do I/O and call external code becomes a turing-complete build system that's hard to sandbox or cache. **Restrict comptime to pure functions over the AST and a controlled stdlib** (like Zig's `comptime`).

**Differentiable types, Slang.D-style.** Bangaru et al.'s `IDifferentiable` interface and `[Differentiable]` attribute, with `fwd_diff` and `bwd_diff` higher-order operators, is the most production-ready autodiff design for a shader language. It compiles to efficient derivative kernels via source-to-source transformation, supports arbitrary control flow, generics, and dynamic dispatch, and outperformed Dr.Jit and Enzyme on the SIGGRAPH benchmarks. Adopt it nearly verbatim ‚Äî the design space here is well-explored and CSSLv3 should not re-invent.

**Information flow control, Jif-style ‚Äî but only as an opt-in capability.** Myers' decentralized label model (DLM) with confidentiality/integrity labels and explicit declassification is the gold standard for security-typed languages. For a graphics language this is overkill, but for **GPU compute used in privacy-sensitive contexts** (federated ML, on-device inference over user data) it's increasingly relevant. Implement IFC as a capability that decorates types ‚Äî `secret<T, alice>` ‚Äî and is checked by the same engine that handles linearity. Don't make every program pay the cognitive cost.

## The rendering vision: radiance cascades over sparse voxel DAGs

CSSLv3 is justified only if it enables rendering techniques that current shading languages make awkward. The most compelling target is the **Sannikov/Freeman radiance cascades pipeline** (2023 Exilecon talk for Path of Exile 2; "Holographic Radiance Cascades for 2D GI", arXiv 2505.02041, May 2025). Radiance cascades store directional radiance in a hierarchy of probes where each level trades angular resolution for spatial resolution at a constant memory budget ‚Äî a 2D implementation runs at 1.85ms for 512√ó512 on an RTX 3080 Laptop. The 3D variant requires aggressive sparse-data-structure traversal, which is exactly what current GLSL/HLSL handle poorly: probe-cascade indexing, ray-interval merging across levels, and atomic accumulation into bricks all benefit from CSSLv3's combination of bindless buffer-device-address pointers, strong typing for grid indices vs. world coordinates, and comptime-generated specialized variants per cascade level.

The geometry side wants **sparse voxel directed acyclic graphs (SVDAGs)**, K√§mpe et al. 2013 with the symmetry-aware (SSVDAG, Villanueva 2016) and transform-aware (TSVDAG, ACM 2025) extensions. SVDAGs deduplicate identical octree subtrees, achieving 10‚Äì50√ó memory reduction over sparse voxel octrees on natural scenes, and traverse with the same ray-marching algorithm. The Aokana paper (2025) demonstrates SVDAG with LOD and streaming for tens of billions of voxels in open-world games. CSSLv3's value here is **type-safe pointer encoding** for SVDAG nodes (the bottom 3‚Äì5 bits of each pointer encode child-reorder transformations and translations) ‚Äî the kind of bitfield discipline that's error-prone in HLSL and natural in a language with first-class linear types.

For dynamic content, **Teardown's voxel approach** (Dennis Gustafsson, Tuxedo Labs) is the proven path: thousands of small axis-aligned voxel volumes rather than one big grid, dense 8-bit-palette storage in v1 evolving to sparse 8¬≥ chunks tracked by a 3D bitmap in the next-gen engine, voxel-vs-voxel CPU collision with SIMD/multithreading, full GPU ray-traced lighting (no triangles except water and power lines). Gustafsson's 2024 next-gen renderer uses Vulkan hardware ray tracing with intersection shaders for voxels, removing shadow-volume limitations. **CSSLv3's linear types are a perfect match for voxel-volume ownership** ‚Äî each volume is a linear resource that can be borrowed mutably by physics or immutably by rendering, with the type system preventing the data races that traditional engines guard against with locks.

For surface/painterly composition, **Alex Evans's Dreams approach** (SIGGRAPH 2015, "Learning from Failure") ‚Äî operationally-transformed CSG trees of signed distance fields evaluated on-the-fly to dense multi-resolution point clouds, then splatted with compute shaders rather than rasterized ‚Äî points toward the long-term endgame. Evans's key insight, after rejecting marching cubes (manifold issues), gigavoxel hybrids (slow), and froxel-LPV (looked too conventional), was that **point splats with imperfect shadow maps gave the painterly aesthetic the team actually wanted**. This is a renderer architecture that demands the kind of GPU-driven, compute-only pipeline that bindless + buffer-device-address makes natural ‚Äî and that current shading languages express clumsily.

For fluids and smoke, **Stam's 1999 Stable Fluids** algorithm (semi-Lagrangian advection + implicit diffusion + Helmholtz-Hodge projection) remains the foundation, with MacCormack and BFECC schemes improving advection accuracy. CSSLv3's comptime can specialize the Poisson solver per grid resolution, and its differentiable-shader support enables **inverse fluid problems** ‚Äî recovering initial conditions or boundary forces from observed flow ‚Äî which Slang.D has shown is tractable.

## Risks, timelines, and the Jai cautionary tale

The dominant risk is **schedule, not technology**. Jai entered closed beta in 2019/2020 and as of April 2026 remains invite-only, with a public release tied to the launch of Blow's *Order of the Sinking Star* in 2026. Twelve years from first demo (2014) to public availability is a project-killing timeline for anything other than a one-person passion project ‚Äî and Jai survived only because Blow self-funded with Braid/Witness/Sokoban revenue. **Odin and Zig, both downstream of Jai's ideas, won the mindshare race by shipping early and accepting design imperfection.** Zig's stage1‚Üístage2‚Üístage3 transition (2016‚Äì2024) is the realistic timeline for a "small" systems language with one full-time lead and a community of contributors.

Concrete failure modes documented across these projects:

- **Bootstrap reproducibility.** Zig now ships a `zig1.wasm` blob in source control to break the chicken-and-egg of "Zig is needed to build Zig" ‚Äî Motiejus Jak≈°tys's 2024 reproducibility analysis showed the blob is faithful to the repo source, but the design is controversial (some prefer a separate non-Zig interpreter). CSSLv3 should commit to **multi-stage bootstrap from C99 from day one** and never accept a binary-only stage.
- **The "every feature is one PhD" trap.** Linear types: 1 PhD. Effect handlers: 1 PhD. Autodiff for shaders: 1 PhD. SPIR-T-style structurization: 1 PhD. Capability-based information flow: 1 PhD. **CSSLv3 needs to ship the first three and defer the last two**, or it will never escape research-prototype status.
- **The shader debugging wall.** Slang's RenderDoc integration via NonSemantic.Shader.DebugInfo.100 took years to mature. Plan for 6 months of debug-tooling investment before the first external user.
- **Driver bug surface.** Intel Arc A770's Vulkan driver, while greatly improved since 2023, still has corner cases around bindless and buffer-device-address that NVIDIA/AMD don't share. CI must run against Mesa/ANV and Windows/Arc separately.

A **realistic 24-month plan**:

| Months | Milestone | Risk if missed |
|---|---|---|
| 0‚Äì6 | HIR + MIR + Cranelift backend; "hello world" runs | Project never starts |
| 6‚Äì12 | SPIR-V backend; first compute shader runs on Arc A770 | No external interest |
| 12‚Äì18 | Linear types + capabilities; Slang.D-style autodiff | Becomes another Odin clone |
| 18‚Äì24 | Effect handlers; comptime metaprogramming; first vertical-slice renderer (radiance cascades over SVDAG) | Yet another unreleased Jai |

## Conclusion: ship the vertical slice, then earn the language

CSSLv3 is most likely to succeed if it is framed not as "a better shading language" or "a better systems language" but as **the language that makes Sannikov-class GI and Gustafsson-class voxel physics expressible by ordinary rendering engineers**. The PL features (linear types, effects, autodiff, comptime) are means, not ends. The SPIR-V/x86 dual backend is a means, not an end. The end is a 60fps scene combining radiance cascades, SVDAG geometry, voxel destruction, and stable-fluids smoke, written in 5,000 lines of one language with no `unsafe` blocks and a 2-second incremental compile.

The reason no shipping language offers this combination today is not that it's impossible ‚Äî every component has been demonstrated separately ‚Äî but that **assembling them requires a team willing to defer features until a vertical slice runs**. Slang built credibility by shipping for Falcor and Unreal before adding autodiff. Koka built credibility by shipping benchmarks against multicore OCaml before adding named handlers. Austral built credibility by shipping a 100-page spec and a working bootstrap compiler before promoting linear capabilities. Jai built no credibility because it shipped nothing, and twelve years later its conceptual children (Odin, Zig) own the mindshare.

The single most important strategic decision is **what to defer**. Defer information flow control to v2. Defer the bespoke x86 backend to v3. Defer Pony-style fine-grained capabilities. Defer macro hygiene formalisms. Defer cross-platform GPU support (start Vulkan-only). Ship the vertical slice in eighteen months with three features done well rather than ten features done halfway, and CSSLv3 has a path to becoming the third member of the Slang/Mojo cohort ‚Äî the languages that earned the right to redefine how their domain is programmed, by first demonstrating they could do the existing job better than the incumbent.
